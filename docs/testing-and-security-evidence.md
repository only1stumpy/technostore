## Реализованные механизмы защиты

| Механизм | Назначение |
| --- | --- |
| JWT в httpOnly cookie | Снижает риск доступа к токену из клиентского JavaScript |
| Проверка актуальной роли из БД | Не позволяет полагаться только на устаревшую роль в JWT |
| Zod-валидация API | Проверяет структуру и значения входных данных |
| Origin validation | Отклоняет mutation-запросы с посторонних сайтов |
| Rate limiting SMS | Ограничивает отправку и проверку кодов по телефону и IP |
| Prisma transactions | Обеспечивает атомарность оформления и отмены заказа |
| Idempotency key и fingerprint | Защищают от повторного создания заказа |
| Условное обновление остатков | Предотвращает продажу товара сверх доступного остатка |
| Security headers и production HSTS | Усиливают защиту браузерных ответов |
| Soft delete и административный аудит | Сохраняют историю данных и административных изменений |
| Безопасный `callbackUrl` | Предотвращает open redirect после авторизации |

## Найденные и исправленные дефекты

| Дефект | Причина | Исправление |
| --- | --- | --- |
| Cursor ломался на значениях с двоеточиями | Ненадёжное разделение строки | Использованы JSON и base64url с поддержкой старого формата |
| Повторная отправка checkout могла создать второй заказ | Не было полной идемпотентности | Добавлены idempotency key, fingerprint и уникальный индекс |
| Конкурентный P2002 обрабатывался внутри ошибочной транзакции | После PostgreSQL-ошибки транзакция может быть aborted | Поиск победившего заказа вынесен после отката транзакции |
| Mutation API имели разрозненную обработку ошибок | Каждый route формировал ответ отдельно | Добавлены общие API helpers и единый формат ошибок |
| Использовались разные параметры возврата после login | Встречались `returnTo`, `redirect` и прямые ссылки | Используется единый безопасный `callbackUrl` |
| Build запускал seed и мог изменять production БД | Seed был частью команды build | Seed вынесен в отдельную команду `db:seed` |
| Недоступная БД могла ломать sitemap во время build | Sitemap всегда запрашивал динамические данные | Добавлен fallback на статические маршруты |

## Ограничения и направления развития

Полный integration-набор с отдельными PostgreSQL и Redis, Playwright E2E, строгая блокировка запросов без `Origin`/`Referer`, HTML CSP, полный accessibility-аудит, нагрузочное тестирование и production monitoring остаются направлениями дальнейшего развития.
